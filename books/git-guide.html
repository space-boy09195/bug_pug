COMMIT MESSAGES

The commit message is a letter to your future self, written at 2am, to be read at 10am the following Tuesday when everything breaks and you cannot remember why you made any of these decisions.

❌ Bad commit messages:

git commit -m "fix"
git commit -m "stuff"
git commit -m "asdfghjkl"
git commit -m "it works now I think"

✅ Good commit messages:

git commit -m "fix: resolve null pointer on empty cart checkout"
git commit -m "feat: add dark mode toggle to settings panel"
git commit -m "refactor: extract payment logic to PaymentService"

BRANCHES

A branch is a parallel universe in which your feature works perfectly and has no merge conflicts. This universe rarely materialises. Still, naming your branches clearly is an act of hope.

git checkout -b feat/user-authentication
git checkout -b fix/login-redirect-loop
git checkout -b chore/update-dependencies
"Never commit directly to main. Unless you need to. Which you never should. But sometimes you do. This is the way."

THE MERGE CONFLICT

You will know a merge conflict by its markings. The ancient symbols appear in your files:

<<<<<<< HEAD Your version of the truth ======= Their version of the truth >>>>>>> feature/their-branch

This is not a technical problem. This is a philosophical problem. Two realities exist simultaneously. Only one can survive. Choose wisely. Then choose git add . and pretend you knew what you were doing.

EMERGENCY COMMANDS

For when things go wrong, and they will go wrong:

git log --oneline # see what happened
git diff HEAD~1 # see what changed
git stash # hide the evidence temporarily
git revert HEAD # undo the last commit safely
git reflog # find commits you thought were lost
"git reflog has saved more careers than any certification." — Anonymous Senior Engineer
